## 1️⃣ DART Collector 확장 검토

### 현재 상태

**활용 중인 데이터**:
- 공시 유형: **`pblntf_ty='B'` (주요사항보고)만 수집**
- 키워드: M&A, 타법인 출자, 신규시설투자, 유상증자 (Phase 1 완료)

**활용률**: ⭐⭐⭐ (60%) — DART API 전체 공시 유형 중 1개만 사용 중

### 🔥 확장 기회 분석

#### 확장 A: 정기공시(A) — R&D 투자 데이터 수집

**데이터 소스**: `pblntf_ty='A'` (정기공시 - 사업보고서, 분기보고서, 반기보고서)

**수집 가능 데이터**:
- **연구개발비**: 삼성전자 반도체 R&D 3조원, SK하이닉스 AI 반도체 개발비 1.5조원 등
- **설비투자 계획**: 제조업체의 CAPEX 계획 (시설투자보다 더 구체적)
- **해외 자회사 출자 현황**: 글로벌 M&A 추적 가능
- **대규모 경상개발비**: 제약·바이오 기업의 신약 개발 투자

**비즈니스 임팩트**:
- 주요사항보고(B)는 **"이미 결정된 투자"**만 잡지만, 정기공시(A)는 **"향후 3~5년 투자 계획"**까지 포함
- 반도체·배터리·바이오 등 **R&D 집약 산업의 투자 트렌드 예측 가능**

**구현 복잡도**:
- 🟡 **중간** — 사업보고서는 XML 구조가 복잡하지만, `dart-fss` 라이브러리가 일부 지원
- `corp.get_document().find_all(class_="연구개발비")` 같은 방식으로 특정 항목 추출 가능

**ROI 추정**:
- 수집량 증가: **현재 대비 +80~120%** (분기보고서는 분기당 2,000건 이상 공시)
- LLM 파싱 비용 증가: 사업보고서 1건당 $0.001~0.003 (본문 길이 ↑)
- **가치**: R&D 투자는 "미래 산업 방향"의 선행 지표 → Silver Layer에서 산업 트렌드 대시보드 구축 시 핵심 데이터

---

#### 확장 B: 지분공시(D) — 대주주 지분 변동 추적

**데이터 소스**: `pblntf_ty='D'` (지분공시 - 5% 이상 대량보유 상황보고)

**수집 가능 데이터**:
- **기관투자자의 지분 매입/매도**: "미래에셋자산운용, 삼성전자 지분 5.2% → 4.8% 감소"
- **벤처캐피탈의 IPO 전 엑시트**: "소프트뱅크벤처스, A사 지분 12% → 3% 매각"
- **적대적 M&A 시그널**: "사모펀드 B, C사 지분 5% 돌파 신고"

**비즈니스 임팩트**:
- 주요사항보고(B)는 회사가 "투자하는" 행위만 잡지만, 지분공시(D)는 **외부 투자자가 "빠지는" 행위**까지 추적 가능
- **기관투자자의 포트폴리오 변화 = 자본 흐름의 역방향 추적**

**구현 복잡도**:
- 🟢 **낮음** — 지분공시는 주요사항보고(B)와 구조가 거의 동일
- `pblntf_ty='D'`로 파라미터만 바꾸면 기존 Collector 재사용 가능

**ROI 추정**:
- 수집량 증가: **현재 대비 +40~60%** (하루 100~150건 공시)
- 구현 시간: **30분~1시간** (기존 코드 복사 후 키워드만 조정)
- **가치**: "누가 빠지고 있는가"는 "누가 들어오는가"만큼 중요한 시그널

---

#### 확장 C: Phase 3 (investment_amount 자동 추출) — 미구현 고급 기능

**현재 문제**:
- `investment_amount` 필드가 모두 `None`으로 저장되어, Silver Layer에서 LLM으로 재파싱 필요
- "SK하이닉스, 용인 반도체 클러스터 3조 투자" 같은 헤드라인에서 **"3조"를 Bronze 단계에서 수치화하지 못함**

**해결 방안 (Phase 3 가이드에 명시)**:
1. **Open DART 주요사항보고서 전용 API** 직접 호출
   - `/api/piicpe_get.json` (타법인 출자 전용) → `inv_amt` 필드 추출
   - `/api/neofac_get.json` (신규시설투자 전용) → `fci_amt` 필드 추출
   - `/api/pifric_decsn.json` (유상증자 전용) → `isu_dcrs_de_fce_amt` 필드 추출

2. **대안**: Silver Layer에서 Gemini Flash로 제목/본문 → `투자금액: XXX원` 추출 (비용: $10~30/10만건)

**비즈니스 임팩트**:
- Bronze Layer에서 금액을 숫자로 보유하면 → **Silver Layer LLM 비용 제로화**
- SQL 쿼리만으로 "2026년 반도체 산업 투자액 TOP 10" 추출 가능

**구현 복잡도**:
- 🔴 **높음** — Open DART API 문서가 불명확하고, `aiohttp` 기반 비동기 병렬 호출 필요
- 예상 소요 시간: **2~4시간**

**ROI 추정**:
- 구현 비용: 2~4시간 개발 + 테스트
- 절감 효과: Silver Layer LLM 파싱 비용 월 $10~30 절감 + 쿼리 성능 대폭 향상
- **가치**: Gold Layer에서 "산업별 투자 규모 추이" 대시보드 구축 시 필수

## 2️⃣ SMES Collector 확장 검토

### 🔥 확장 기회 분석

#### 확장 A: 선정 결과 API — "누가 실제로 지원금을 받았는가"

**문제 인식**:
- 현재는 **"공고"**만 수집 → 1,000개 기업이 지원했는데 20개만 선정되는 경우, 선정 기업 정보 없음
- 비즈니스 관점: **"정부가 어떤 기업을 선택했는가" = 유망 스타트업 조기 발굴 시그널**

**예상 API**:
- `mssBizService_v2/getSelectionResult` (추정 — 실제 API 명세 확인 필요)
- 응답 예상 필드:
  - `pblancId`: 공고 ID (기존 공고와 JOIN 가능)
  - `slctnEntrpsNm`: 선정 기업명
  - `slctnAmt`: 선정 금액 (실제 지급액)
  - `slctnDt`: 선정 발표일

**비즈니스 임팩트**:
- **"정부 지원금 받은 스타트업" = 투자자 관심 기업**
- "2025년 AI 분야 정부 지원 TOP 50 기업" 리스트 자동 생성 가능
- `raw_economic_data`와 교차 분석: "정부 지원 → 6개월 후 VC 투자 유치" 패턴 발견

**구현 복잡도**:
- 🟡 **중간** — API 신청 필요 (SMES 포털에서 동일 인증키로 추가 신청 가능성 높음)
- 기존 `smes_collector.py` 패턴 재사용 가능

**ROI 추정**:
- API 신청 시간: 1~2영업일
- 구현 시간: 1~2시간
- **가치**: "정부가 선택한 기업 = 검증된 스타트업" 필터로 Silver Layer에서 추천 정확도 향상

---

#### 확장 B: 예산 집행 현황 API — "지원금이 실제로 집행되었는가"

**문제 인식**:
- 선정되었어도 **실제 지급까지 3~6개월** 소요
- "선정 발표 ≠ 지급 완료" → 정확한 자본 흐름 추적 불가

**예상 API**:
- `mssBizService_v2/getBudgetExecution` (추정)
- 응답 예상 필드:
  - `pblancId`, `slctnEntrpsNm`
  - `exctnDt`: 집행(지급) 완료일
  - `exctnAmt`: 실제 집행 금액 (선정 금액 ≠ 집행 금액인 경우 많음)

**비즈니스 임팩트**:
- "정부 예산 집행 속도" = 정책 실효성 지표
- "선정 발표 → 실제 지급"까지의 지연 시간 추적으로 정부 지원사업 효율성 평가 가능

**구현 복잡도**:
- 🟡 **중간** — API 존재 여부 확인 필요

**ROI 추정**:
- **가치**: Gold Layer에서 "정부 지원사업 실효성 대시보드" 구축 가능

---

#### 확장 C: 타 부처 공고 API 통합 — 과기부, 산업부 등

**확장 가능성**:
- 현재: 중소벤처기업부만 수집
- 공공데이터포털에는 다음 API도 존재:
  - **과학기술정보통신부 R&D 과제 공고**
  - **산업통상자원부 신산업 지원사업**
  - **고용노동부 청년창업 지원**

**비즈니스 임팩트**:
- 중기부는 "창업 초기" 지원이 강하지만, 과기부는 **"기술 개발 단계"** 지원이 강함
- 전체 정부 지원사업을 통합하면 **"기업의 성장 단계별 지원 히스토리"** 추적 가능

**구현 복잡도**:
- 🟡 **중간** — 각 부처 API 구조가 미묘하게 다를 가능성 (XML 필드명 상이)

**ROI 추정**:
- 수집량 증가: **현재 대비 +200~300%** (타 부처 공고 합치면 하루 500건 이상)
- **가치**: "정부 전체 R&D 예산 흐름" 추적으로 산업 정책 방향 예측 가능

## 3️⃣ Wowtale Collector 확장 검토

### 현재 상태

**활용 중인 데이터**:
- 데이터 소스: **RSS 피드 (`https://wowtale.net/feed/`)만 사용**
- 수집 방식: `feedparser` → 제목 + 요약 + `content:encoded` (전문)
- **활용률**: ⭐⭐⭐⭐ (80%) — RSS가 제공하는 데이터는 거의 모두 활용 중

---

### 🟡 확장 기회 분석 (제한적)

#### 확장 A: 본문 2-Step 크롤링 (Phase 2 미구현)

**현재 한계**:
- `content:encoded` 가 제공하는 본문은 **요약본**인 경우가 많음 (전문 1,000자 vs 실제 기사 3,000자)
- 투자 금액·VC 목록 등이 **본문 하단**에 있는 경우 → RSS만으로는 누락

**개선안**:
1. RSS에서 `entry.link` (기사 원문 URL) 추출
2. `aiohttp`로 원문 페이지 HTML 크롤링
3. `BeautifulSoup`으로 `div.entry-content` (본문 영역)만 추출
4. `raw_metadata["full_content"]`에 전문 저장

**비즈니스 임팩트**:
- Silver Layer LLM이 더 정확한 투자 금액·VC 목록 추출 가능
- "카카오벤처스, 알토스벤처스, 캡스톤파트너스 공동 투자" 같은 **복수 VC 리스트** 정확히 파싱

**구현 복잡도**:
- 🟡 **중간** — Wowtale 사이트 구조 분석 필요 (WordPress라 `div.entry-content` 클래스는 거의 표준)
- 예상 소요 시간: 1~2시간

**ROI 추정**:
- 수집량 증가: 0% (건수는 동일)
- 데이터 품질 향상: **+30~50%** (LLM 파싱 정확도 상승)
- **가치**: Silver Layer에서 "누가 누구와 함께 투자했는가" (VC 네트워크 분석) 가능

**가능성 낮음**:
- Wowtale은 미디어 사이트라 **별도 API 제공 가능성 낮음**
