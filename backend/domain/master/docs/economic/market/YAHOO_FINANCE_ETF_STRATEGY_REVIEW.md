## 2. 거래량 급증 기준 검토 (2.0배)

### 초안 알고리즘

```python
if volume_ratio >= 2.0:  # 20일 평균 대비 2배
    # Bronze 적재
```

### 🚨 발견된 문제: "고정 임계값의 함정"

**문제점**:
ETF마다 **거래량 변동성**이 다릅니다.

| ETF | 일평균 거래량 | 표준편차 | 2배 급증 빈도 (추정) |
|-----|-------------|---------|-------------------|
| TIGER 글로벌AI액티브 | 500만 주 | 200만 주 (40%) | 월 2~3회 (적절) |
| KODEX K-신재생에너지 | 50만 주 | 30만 주 (60%) | 월 8~10회 (너무 많음) |

→ **고정 2.0배 기준을 쓰면, 변동성 높은 ETF에서 "노이즈 데이터" 과다 수집**

---

### ✅ 개선안 1: 표준편차 기반 동적 임계값

**제안**:
```python
# Step 1: 20일 평균 + 표준편차 계산
avg_volume = hist['Volume'].rolling(20).mean()
std_volume = hist['Volume'].rolling(20).std()

# Step 2: Z-score 계산 (통계적 이상치 탐지)
yesterday_volume = hist.iloc[-1]['Volume']
z_score = (yesterday_volume - avg_volume.iloc[-1]) / std_volume.iloc[-1]

# Step 3: Z-score >= 2.0 이면 "통계적으로 유의미한 급증"
if z_score >= 2.0:
    # Bronze 적재
```

**장점**:
- ETF마다 다른 변동성을 자동으로 고려
- 통계 이론 기반 (정규분포 가정 시 Z=2.0 → 상위 2.5%)

**단점**:
- 구현 복잡도 소폭 증가
- 초보 사용자가 이해하기 어려움

---

### ✅ 개선안 2: 멀티 레벨 임계값 (실용적 절충안)

**제안**:
```python
# ETF별로 임계값을 다르게 설정 (운영 경험 기반 조정)
THRESHOLDS = {
    "091220.KS": 2.0,  # TIGER 글로벌AI액티브 (안정적)
    "441680.KS": 2.0,  # KODEX 2차전지산업 (안정적)
    "244620.KS": 2.5,  # KODEX 바이오 (변동성 높음)
    "261140.KS": 2.5,  # KODEX K-신재생에너지 (변동성 높음)
    "332620.KS": 2.2,  # TIGER K-푸드 (중간)
}

ticker = "091220.KS"
threshold = THRESHOLDS.get(ticker, 2.0)  # 기본값 2.0

if volume_ratio >= threshold:
    # Bronze 적재
```

**장점**:
- 구현 간단 (설정 파일 방식)
- 운영하면서 티커별로 임계값 튜닝 가능
- 이해하기 쉬움

**단점**:
- 수동 튜닝 필요 (초기 설정값은 추정)

## 4. `investment_amount` 계산 검토

### 초안: `거래량 × 종가`

```python
inflow_amount = int(yesterday_volume * yesterday_close)
```

### 🚨 발견된 문제: "종가 편향"

**문제점**:
ETF 거래는 하루 종일 발생하는데, 종가만 쓰면 **실제 유입 금액과 괴리**

**사례**:
```
TIGER 글로벌AI액티브 (2026-05-11)
- 시가: 10,000원
- 고가: 11,000원
- 저가: 9,500원
- 종가: 9,800원
- 거래량: 1,000만 주

초안 계산: 1,000만 주 × 9,800원 = 980억원
실제 유입: 거래 대부분이 10,500원 부근 → 약 1,050억원
오차: 약 70억원 (7%)
```

---

### ✅ 개선안: VWAP (Volume Weighted Average Price) 사용

**제안**:
```python
# yfinance는 VWAP을 직접 제공하지 않음
# → (고가 + 저가 + 종가) / 3 으로 근사 (HLCC/3)

approx_avg_price = (high + low + close) / 3
inflow_amount = int(yesterday_volume * approx_avg_price)
```

**정확도 비교**:
| 방식 | 오차 | 구현 난이도 |
|------|------|------------|
| 종가만 | ±5~10% | 쉬움 |
| (고+저+종)/3 | ±2~5% | 쉬움 |
| 진짜 VWAP | ±1% | 어려움 (틱 데이터 필요) |

**권장**: **(고+저+종)/3** — 오차 절반 감소 + 구현 간단

## 5. `yfinance` 리스크 대응 검토

### 초안: 버전 고정 (`yfinance==0.2.40`)

**문제**: 버전 고정만으로는 **Yahoo 정책 변경을 막을 수 없음**

---

### ✅ 개선안: 3단계 Fallback 전략

```python
class YahooFinanceCollector:
    async def collect(self):
        try:
            # Plan A: yfinance (1순위)
            return await self._collect_via_yfinance()
        except Exception as e:
            logger.error(f"yfinance 실패: {e}")
            
            try:
                # Plan B: 네이버 금융 크롤링 (2순위)
                return await self._collect_via_naver()
            except Exception as e2:
                logger.error(f"네이버 금융 실패: {e2}")
                
                # Plan C: 빈 리스트 반환 (다른 Collector는 정상 동작)
                return []
    
    async def _collect_via_naver(self):
        """네이버 금융에서 동일 ETF 데이터 크롤링 (Fallback)."""
        # 예: https://finance.naver.com/item/main.nhn?code=091220
        pass
```

**장점**:
- yfinance가 깨져도 파이프라인 전체가 멈추지 않음
- 네이버 금융은 한국 ETF에 더 안정적

**단점**:
- Plan B 구현 추가 시간 (1~2시간)
- 네이버도 구조 변경 가능

**권장**: Phase 1에서는 Plan A만 구현, Plan B는 **yfinance 장애 발생 시 긴급 대응**

## 6. Bronze → Silver 연계 검토

### 초안: Bronze에 "거래량 급증" 이벤트만 저장

**문제**: Silver Layer에서 **"왜 급증했는지"** 알 수 없음

**사례**:
```
Bronze 데이터:
{
  "source_type": "YAHOO_ETF_AI",
  "raw_title": "TIGER 글로벌AI액티브 거래량 2.3배 급증 (2026-05-11)",
  "investment_amount": 1050000000000,
}

Silver Layer의 고민:
- 이게 "정부 정책 발표" 때문인가?
- 아니면 "삼성전자 실적 호조" 때문인가?
- 단순 "투기 자금 유입"인가?
```

→ **DART / RSS 뉴스와 교차 검증이 어려움**

---

### ✅ 개선안: `raw_metadata`에 당일 뉴스 스냅샷 추가

**제안**:
```python
# Collector에서 거래량 급증을 탐지했을 때,
# 해당 테마 관련 뉴스를 간단히 검색해서 메타데이터에 추가

raw_metadata = {
    "ticker": "091220.KS",
    "volume_ratio": 2.3,
    # ... (기존 필드) ...
    
    # 🆕 추가: 당일 관련 뉴스 (최대 3건)
    "related_news_snapshot": [
        {
            "headline": "정부, AI 반도체 1조원 투자 발표",
            "source": "Wowtale",
            "published_at": "2026-05-11T10:00:00+09:00",
        },
        {
            "headline": "삼성전자, AI 칩 생산 확대",
            "source": "DART",
            "published_at": "2026-05-11T09:30:00+09:00",
        },
    ],
}
```

**구현 방식**:
- Bronze Collector가 **같은 날 수집된 DART/RSS 데이터를 조회**
- 테마 키워드 매칭 (AI ETF → "AI", "반도체", "칩" 등)
- 최대 3건만 메타데이터에 첨부

**장점**:
- Silver Layer에서 **"원인 분석"**이 훨씬 쉬워짐
- 대시보드에서 "거래량 급증 + 관련 뉴스" 동시 표시 가능

**단점**:
- Collector 복잡도 증가
- Bronze 테이블 간 의존성 발생 (설계 원칙 위배 가능성)

**권장**: Phase 2로 미루고, Phase 1에서는 **Silver Layer가 수동으로 교차 검증**
