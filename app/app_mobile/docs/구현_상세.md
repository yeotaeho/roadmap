# app_mobile 구현 상세

본 문서는 리포지토리 내 `app/app_mobile`에 적용된 **Flutter 초기 세팅·아키텍처 스캐폴딩**을 코드 기준으로 기술합니다. Next.js 웹(`www.yeotaeho.kr`)의 **4개 메인 탭 경로**와 맞추고, FastAPI 백엔드 연동을 염두에 둔 구조입니다.

---

## 1. 요약

| 항목 | 내용 |
|------|------|
| **패키지명** | `app_mobile` (`pubspec.yaml`의 `name`) |
| **표시 제목** | `청년 인사이트` (`MaterialApp.router`의 `title`) |
| **루트 경로** | 리포지토리 기준 `app/app_mobile` |
| **아키텍처 스타일** | Feature-First: `lib/features/<도메인>/presentation/` |
| **상태 관리** | `flutter_riverpod` + `hooks_riverpod` (루트에 `ProviderScope`) |
| **라우팅** | `go_router` — `StatefulShellRoute.indexedStack` + 하단 `NavigationBar` |
| **HTTP** | `dio` + `AuthInterceptor` — `TokenStorage.readAccessToken`과 연결 (`dioProvider`) |
| **환경 변수** | `flutter_dotenv` + **리포지토리 루트** `docker-compose.yaml` 옆 `.env` (IO에서 파일 읽기) + `--dart-define` 오버라이드 |
| **UI 데이터 시각화·텍스트** | `fl_chart` (미연동), `flutter_markdown_plus` — AI 코치 샘플 `MarkdownBody` 사용 |
| **모델 코드 생성** | `freezed` + `json_serializable` + `build_runner` (dev) |

## 2. 프로젝트 생성

### 2.1 실행한 명령

프로젝트는 Monorepo의 **`app` 디렉터리 아래**에 두었습니다.

```bash
cd <repo>/app
flutter create --org kr.yeotaeho app_mobile --platforms=android,ios
```

- **`--org kr.yeotaeho`**: Android `applicationId` / iOS 번들 ID 접두에 사용됩니다.

### 2.2 앱 식별자 (번들 / 패키지)

| 플랫폼 | 값 | 비고 |
|--------|-----|------|
| **Android** | `kr.yeotaeho.app_mobile` | `android/app/build.gradle.kts`의 `applicationId` |
| **iOS** | `kr.yeotaeho.appMobile` | Flutter 기본 규칙(마지막 세그먼트 camelCase). `ios/Runner.xcodeproj/project.pbxproj`의 `PRODUCT_BUNDLE_IDENTIFIER` |

웹·백엔드 문서와 완전히 동일한 문자열을 쓰려면 iOS 쪽 번들 ID를 수동으로 `kr.yeotaeho.app_mobile` 형태로 맞출 수 있습니다(현재는 생성기 기본값 유지).

### 2.3 Dart SDK

- `pubspec.yaml` → `environment.sdk: ^3.10.8`
- 로컬 개발 시 Flutter 안내에 맞는 Dart 버전 사용 권장.

## 5. 진입점: `main.dart`

- `WidgetsFlutterBinding.ensureInitialized()` 후 **`await AppEnv.load()`** — (모바일·데스크톱) `Directory.current` 기준 상위 디렉터리를 탐색해 `docker-compose.yaml`과 같은 폴더의 `.env`를 읽습니다.
- 최상위를 **`ProviderScope`**로 감싸 Riverpod 사용 준비.
- **`YouthInsightApp`**: `MaterialApp.router`  
  - `theme` / `darkTheme` / `themeMode: ThemeMode.system` — 시스템 라이트·다크 전환.
  - `routerConfig: appRouter` — `go_router` 단일 진입.
  - `debugShowCheckedModeBanner: false`.

웹의 `class` 기반 다크 모드와 달리, Flutter는 **시스템(UI 모드)** 기준입니다. 앱 내 수동 토글을 넣으려면 `themeMode`를 `StateProvider` 등으로 바꾸면 됩니다.

## 6. 라우팅: `core/router/app_router.dart`

### 6.1 설계 의도

- Next.js **메인 탭**과 동일한 **경로 문자열**을 유지해 기획·QA·딥링크 문서를 공유하기 쉽게 함.

| 인덱스 | 경로 | `GoRoute` name | 화면 위젯 |
|--------|------|----------------|-----------|
| 0 | `/` | `dashboard` | `DashboardPage` |
| 1 | `/consult` | `consult` | `ConsultPage` |
| 2 | `/roadmap` | `roadmap` | `RoadmapPage` |
| 3 | `/coach` | `coach` | `CoachPage` |

### 6.2 기술 요약

- **`StatefulShellRoute.indexedStack`**: 탭 전환 시 각 브랜치 **스택을 유지** (다시 탭해도 스크롤 위치 등 유지에 유리).
- **`StatefulNavigationShell`**: `body`에 그대로 두고, `currentIndex`와 `goBranch(index)`로 탭 전환.
- **`goBranch` 두 번째 인자** `initialLocation: index == navigationShell.currentIndex`: **같은 탭 재선택 시** 해당 브랜치 초기 위치로 리셋하는 패턴에 사용 (필요 시 동작 조정 가능).
- 각 탭 **`NoTransitionPage`**: 탭 전환 시 불필요한 페이지 애니메이션 억제.

### 6.3 하단 내비: `MainTabScaffold`

- 구 예제의 `BottomNavigationBar` 대신 **Material 3 `NavigationBar`** 사용 (`indicatorColor`로 선택 막대 강조).
- 라벨: `인사이트`, `상담`, `로드맵`, `코치` (웹 `MainTabBar` 라벨과 대응).

### 6.4 전역 네비게이터 키

- `rootNavigatorKey`: 추후 다이얼로그·풀스크린 라우트에서 `Navigator` 참조용으로 확장 가능.

## 8. 환경 변수 `API_BASE_URL` (`core/config/app_env.dart`)

**우선순위 (높은 것이 이김):**

1. **`--dart-define=API_BASE_URL=...`** — CI·임시 오버라이드·스토어 빌드에 적합.
2. **Monorepo 루트 `.env`** — `docker-compose.yaml`(또는 `docker-compose.yml`)과 **같은 디렉터리**에 있는 `.env` 파일. `flutter run`/`flutter test` 시 작업 디렉터리에서 위로 올라가며 탐색합니다 (`root_env_loader_io.dart`).
3. 둘 다 없으면 **`https://api.example.com`**.

로드 실패 시에도 `AppEnv.apiBaseUrl`은 안전하게 위 기본값으로 동작합니다(`dotenv.isInitialized` 검사).

**파일·동작**

| 구분 | 설명 |
|------|------|
| 리포지토리 루트 `.env` | Docker·Next.js와 **동일 파일**. `API_BASE_URL` 키를 추가하면 앱이 사용합니다. |
| `app_mobile/.env.example` | 키 이름 안내용 샘플(루트 `.env`에 넣을 변수 설명). |
| Web 타깃 | `dart:io` 미사용 → stub만 적용, 파일 미로드. `dart-define` 또는 호스트 설정에 의존. |

**주의:** 루트 `.env`에는 DB·OAuth 시크릿 등이 들어갈 수 있습니다. 앱 프로세스에는 **전체 파일 내용**이 dotenv 맵으로 적재되므로, 릴리스 빌드에서는 루트 `.env`가 단말에 없거나 **`dart-define`만 쓰는** 편이 안전합니다.

## 9. 인증 저장소·Dio 연동 (`core/auth`, `core/providers`)

| 구성 요소 | 역할 |
|-----------|------|
| `TokenStorage` | `access_token`, `refresh_token` 키로 secure storage 읽기/쓰기·세션 삭제. |
| `tokenStorageProvider` | 앱 전역에서 동일 `TokenStorage` 인스턴스 주입. |
| `dioProvider` | `DioClient.create(baseUrl: AppEnv.apiBaseUrl, getAccessToken: storage.readAccessToken)`. |

로그인 API 연동 후 **`writeAccessToken`**(및 필요 시 **`writeRefreshToken`**)만 호출하면, 이후 모든 Dio 요청에서 인터셉터가 자동으로 `Authorization: Bearer …`를 붙입니다. 로그아웃 시 **`clearSession`**.

## 10. 네트워크: `dio_client.dart`, `interceptors.dart`

### 10.1 `DioClient.create`

- **`baseUrl`**: 호출부에서 **`AppEnv.apiBaseUrl`** 전달 (`dioProvider`).
- **`getAccessToken`**: 필수 — `TokenStorage.readAccessToken` 연결.
- **타임아웃**: 연결 15초, 수신 30초.
- **헤더**: `Accept: application/json`.
- **인터셉터 순서**: `AuthInterceptor` → (디버그만) `LogInterceptor`.

### 10.2 `AuthInterceptor`

- **매 요청마다** `getAccessToken()` 호출 → 토큰이 비어 있으면 헤더 생략.
- 401 처리·리프레시 토큰 로테이션은 **`onError`**에서 확장 예정.

### 10.3 로깅

- `LogInterceptor`는 **`kDebugMode`일 때만** 등록 — 릴리스 빌드에서 요청/응답 본문 로그 유출 방지.

## 13. 테스트

### 13.1 `test/widget_test.dart`

- `TestWidgetsFlutterBinding.ensureInitialized()` 및 **`await AppEnv.load()`** 후 위젯 트리 기동(에셋 로드와 실제 앱과 정렬).

- `flutter test` 재시도 또는 Flutter SDK 업데이트
- CI에서는 Linux/macOS 러너에서 테스트

## 16. 웹(Next.js)과의 대응 표

| 웹 (`www.yeotaeho.kr`) | Flutter (`app_mobile`) |
|------------------------|-------------------------|
| `MainLayout` + `MainTabBar` | `MainTabScaffold` + `NavigationBar` |
| `/`, `/consult`, `/roadmap`, `/coach` | 동일 경로 `GoRoute` |
| `.env`(Monorepo 루트) | `root_env_loader_io` + `AppEnv.apiBaseUrl` |
| Zustand 등 | Riverpod (`ProviderScope`, `dioProvider` 등) |
| Axios + 인터셉터 | Dio + `AuthInterceptor` + `TokenStorage` |
| `dark` 클래스 토글 | `ThemeMode.system` (앱 내 토글은 미구현) |
| Recharts | `fl_chart` (미연동) |
| 마크다운(코치/아카이브) | `flutter_markdown_plus` (`Markdown`, `MarkdownBody`) |
