# 기술 아키텍처 전략: SimpleDiscoveryClient

## 개요

본 프로젝트는 1인 개발 환경 및 MVP 단계에서 가장 효율적인 마이크로서비스 아키텍처 전략으로 **SimpleDiscoveryClient**를 채택했습니다.

## 전략 비교

### Eureka vs SimpleDiscoveryClient

|구분|Eureka|SimpleDiscoveryClient|
|---|---|---|
|**복잡도**|높음 (별도 서버 운영 필요)|낮음 (설정 파일 기반)|
|**리소스 사용**|높음 (추가 서버 인스턴스)|낮음 (설정 파일만 사용)|
|**관리 용이성**|중간 (서버 모니터링 필요)|높음 (설정 파일만 관리)|
|**로드밸런싱**|자동|설정 기반|
|**헬스체크**|자동|제한적 (향후 K8s로 해결)|

## SimpleDiscoveryClient 채택 이유

### 1. 현재 프로젝트 환경에 최적화

- **서비스 개수가 적음**: MVP 단계에서는 서비스 수가 제한적
- **Docker Compose 환경**: 로컬 개발 및 테스트 환경에 적합
- **1인 개발**: 복잡한 인프라 관리 부담 최소화

### 2. 효율성

- **리소스 절약**: 별도 서비스 디스커버리 서버 불필요
- **빠른 개발**: 설정 파일 기반으로 즉시 적용 가능
- **낮은 복잡성**: 학습 곡선 최소화

### 3. 단점 보완 전략

**제한적인 헬스체크:**
- 현재: 설정 파일 기반 서비스 등록
- 향후: Kubernetes(K8s) 도입 시 Readiness Probe로 해결
- 단기: 수동 모니터링으로 충분

## 마이그레이션 전략

### 단기 (MVP 단계)
- ✅ SimpleDiscoveryClient 사용
- ✅ Docker Compose 환경
- ✅ 설정 파일 기반 서비스 등록

### 중기 (서비스 확장)
- 🔄 서비스 개수 증가 시 고려
- 🔄 헬스체크 자동화 필요 시 검토

### 장기 (프로덕션 확장)
- 🚀 Kubernetes(K8s) 도입
- 🚀 K8s Service Discovery 활용
- 🚀 Readiness/Liveness Probe로 헬스체크 자동화

## 결론

**SimpleDiscoveryClient**는 현재 프로젝트 규모 및 환경에 가장 적합하며, 가장 효율적인 전략입니다. 단기 전략으로 채택하고, 향후 확장 계획에 따라 마이그레이션을 진행하기로 결정했습니다.
